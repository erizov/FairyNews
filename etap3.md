## 1. Тема, описание задачи

**Тема:** прототип **нейро-сказочника** (нейро-сотрудника) — системы, которая по выбранной пользователем новостной повестке и выбранному (или шаблонному) сказочному каркасу генерирует **новую сказку на русском языке** в духе русской народной сказки, опираясь на **ретрив контекста из корпуса сказок** (RAG) и краткое представление новости.

**Задача этапа 3:** собрать **работоспособный MVP-прототип** цепочки «новость → контекст → генерация текста», зафиксировать **структуру нейро-сотрудника**, описать **источники RAG** и **технологии сбора данных**, дать **план полного пайплайна** от сырой новости до готовой сказки и **план трёхстраничного интерфейса**. Получить **первые качественные/количественные наблюдения** (примеры выхода, очевидные ограничения).

**Ограничения MVP (явно вне объёма на этом этапе):** без отдельной подсистемы безопасности, без журналирования и отчётности, без учётных записей и входа в систему — только минимально достаточная демонстрация сценария для одного пользователя/сессии.

## 2. База и обработка базы (без конфиденциальных данных)

### 2.1. Что входит в базу знаний для RAG (сказки)

В отчёте можно описать **обезличенно**:

- **Источники полнотекстов:** открытые корпуса (в т.ч. **Project Gutenberg** по ID из конфигурации семян), **локальные UTF‑8 тексты** в иерархии каталогов проекта (русский / советский слой и др.), метаданные: домен, источник, автор, страна, язык контента, опционально подсказки по героям.
- **Обработка:** нарезка на чанки фиксированного размера с перекрытием, вычисление **эмбеддингов** (мультиязычная модель семейства sentence-transformers), хранение в **векторной БД** (**ChromaDB**, локальный каталог данных проекта).
- **Инкремент:** сравнение **хешей** полного текста источника с сохранённым состоянием; при изменении — пересчёт чанков только для затронутых произведений.

*В отчёт не включать:* пути к секретам, внутренние имена серверов, персональные данные.

### 2.2. Новости (не в индексе RAG по сказкам)

Согласно текущей концепции репозитория, **политические новости в векторный индекс сказок не кладутся**; для прототипа новость подаётся как **краткая выжимка / выбор пользователем готового пункта** (см. пайплайн и UI ниже). **Технологии сбора новостей** (план/прототип): RSS и открытые API где доступны; иначе — выборочный HTTP-запрос и разбор HTML при соблюдении правил площадки; нормализация в плоский текст; дедупликация по заголовку/URL; **без публикации сырых лент** в отчёте — только описание метода и типа полей (заголовок, дата, краткий текст).

## 3. Структура нейро-сотрудника (схематично, блоки)

Ниже — логическая схема **без привязки к конкретному фреймворку UI**. Стрелки: поток данных и вызовов.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                         Ввод пользователя (MVP)                          │
│  Стр.1: выбор новости  →  Стр.2: выбор сказки/шаблона  →  Стр.3: текст   │
└─────────────────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  Блок A. Подготовка новостного сигнала                                    │
│  Нормализация выбранной новости в компактный промпт-контекст (факты,      │
│  тема; без хранения «банка новостей» в MVP при ручном выборе).            │
└─────────────────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  Блок B. Запрос к RAG по сказкам                                          │
│  По объединённому запросу (новость + желаемый сказочный каркас/герой)     │
│  извлекаются top-k чанков из Chroma; опционально фильтры по метаданным.   │
└─────────────────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  Блок C. Агент генерации (LLM)                                            │
│  Промпт: стиль русской народной сказки + retrieved фрагменты + новость.  │
│  Выход: связный текст сказки на русском.                                  │
└─────────────────────────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  Блок D. (Опционально в полной версии) аудит / QA                         │
│  В MVP может быть отключён или сведён к одной «самопроверке» в том же    │
│  вызове; в отчёте — как перспектива после этапа 3.                        │
└─────────────────────────────────────────────────────────────────────────┘
```

**Кратко по блокам:**

| Блок | Назначение |
|------|------------|
| A | Сжать новость до безопасного для контекстного окна объёма; убрать разметку; при автосборе — единый формат полей. |
| B | Дать модели **опорные сюжеты и формулировки** из легального корпуса сказок (RAG), не смешивая новости в индекс сказок. |
| C | Основной **нейро-сотрудник** в узком смысле: автор текста. |
| D | Расширение пайплайна для устойчивости и этики — после MVP. |

## 5. Источники RAG и технология сбора данных (сводка для отчёта)

**RAG (сказки):**

| Элемент | Содержание |
|---------|------------|
| Полнотексты | Gutenberg (PD), локальные `.txt`, конфиг семян YAML |
| Чанкование + эмбеддинги | Параметризуемые размер/перекрытие; multilingual MiniLM (как в README проекта) |
| Хранилище | ChromaDB, коллекция под сказки |
| Метаданные | `domain`, `source`, `author`, `country`, `content_lang`, герои и др. |

**Сбор данных (новости, план/описание для отчёта):**

| Элемент | Содержание |
|---------|------------|
| Каналы | RSS/API приоритетно; парсинг HTML при необходимости и допустимости |
| Стек | HTTP-клиент (например httpx), парсер разметки при появлении задачи (BeautifulSoup/lxml) |
| Обработка | Кодировка UTF-8, очистка HTML, дедупликация, краткая аннотация для LLM |

## 6. Пайплайн: от новости до финальной сказки (только план, без обязательной реализации на этапе 3)

*Это пошаговый **проектный** конвейер; реализация может идти итерациями после MVP.*

1. **Источник новости:** RSS/API/ручной список заголовков → запись `{заголовок, дата, url?, краткий текст}`.
2. **Нормализация:** удаление мусорной разметки, лимит длины, язык RU (или пометка языка).
3. **Выбор пользователем:** на экране 1 пользователь указывает **одну** новость (или версию выжимки).
4. **Построение retrieval-запроса:** объединение сущности новости + жанровые ключевые слова + опционально имя героя/мотив со страницы 2.
5. **RAG:** поиск top-k чанков в индексе **только сказок**; при необходимости фильтр по `country` / `domain` / автору.
6. **Сборка промпта:** системная инструкция (стиль русской сказки) + «факты новости» + «цитаты/фрагменты из корпуса» + выбранный **шаблон сказки** или **дефолт** («без имени классики, общий народный каркас»).
7. **Вызов LLM:** генерация полного текста сказки одним или двумя этапами (черновик → полировка — по объёму токенов).
8. **Вывод:** страница 3 — готовый текст; в MVP **без** сохранения в личный кабинет, **без** отчёта на сервере и **без** журнала аудита.

## 7. План пользовательского интерфейса (MVP, три страницы)

| Страница | Содержание |
|----------|------------|
| **Страница 1** | Пользователь **выбирает новость**: список из последних заголовков (мок-данные допустимы в MVP) или поле вставки краткой выжимки; кнопка «Далее». |
| **Страница 2** | Пользователь **выбирает сказочную опору**: из списка (название/герой/источник корпуса) **или** переключатель «**по умолчанию**» — тогда retrieval и стиль задаются без именованного произведения; кнопка «Сгенерировать». |
| **Страница 3** | Отображение **сгенерированной сказки** целиком; опционально кнопка «Назад» без обязательной истории сессий. |

**MVP:** один пользовательский сценарий, без логина, без страницы метрик/отчётов, без админки и без логирования на сервере.

## 8. Выводы

**Результат на текущей итерации (проверено тестами):**

- Реализована связка **новость → структурирование (агент новостей) → RAG по сказкам с выбором якорного произведения по схожести эмбеддингов → генерация сказки → аудит → вопрос с эталонным ответом**.

- Интеграционный e2e `tests/test_e2e_multi_agent.py` при собранном Chroma и **моке LLM** проходит успешно (`2 passed`, один тест с живым API по умолчанию отключён).
- Узкие места: доступность API по региону; качество и объём индекса; длина контекста при большом числе чанков.

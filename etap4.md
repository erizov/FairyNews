## 2. Критерии к ноутбуку (финальная версия)

1. **Friendly user interface:** человеку без знания внутренностей достаточно **просмотреть ячейки с «функциями»** и в **конце** запустить **тестовую ячейку**, где через **`input()`** или **`google.colab.files.upload()`** подаются свои данные и выводится **понятный результат** (текст сказки, краткая сводка по шагам, при необходимости — путь к сохранённому отчёту).
2. **Модульность:** весь код разбит на отдельные части («функции» в широком смысле); есть **одна главная «функция» запуска** (оркестратор), которая вызывает остальные; **последняя ячейка** предназначена для проверки (вызов этой главной функции с тестовыми данными).
3. **Дополнительные файлы:** если нужны база/индекс/YAML — лежат **рядом с ноутбуком** и описаны в README внутри папки или в первой ячейке ноутбука.
4. **Пути:** во всех «функциях», которые **читают/пишут файлы**, пути — **параметры со значениями по умолчанию** (можно запускать без явного пути).
5. **Чистота:** каждая «функция» использует **только** переданные параметры и вызовы других «функций»; **не** читает глобальные результаты предыдущих шагов, если они не переданы аргументами (исключение — вызов другой «функции», возвращающей новые данные внутри текущей).

**Связь с проектом FairyNews (ориентир для финальной версии):** логика **новость → агент новостей → RAG по сюжетам сказок → генерация → аудит → вопрос–ответ** должна быть воспроизводима в ноутбуке в виде явных шагов и главной функции `run_pipeline(...)` (имя условное).

## 3. Критерии к файлу с экспериментами

1. **Структурированный вид** (оглавление, таблицы, единый формат блоков).
2. **Описание экспериментов** — что менялось (промпты, температура, число чанков RAG, пресеты доменов и т.д.) и **каков был результат** (качественно и по возможности метриками).
3. **Полнота:** каждый эксперимент описан целиком, **даже при повторении** формулировок между вариантами.
4. **Один эксперимент — один раздел** (Вариант №1, №2, …).
5. **Итог:** сводка по всем вариантам, **лучший вариант** с обоснованием; достигнута ли цель; если нет — **почему** и **какие ещё пути** (другая модель, больше корпус, иной ретрив, пост-обработка и т.д.).

## 4. E2E-тестовый пайплайн

**Цель:** автоматизированный **сквозной** сценарий: по **выбранной новости** и **RAG по сюжетам сказок** получить **сказку в русском народном стиле**, сходную по смыслу с новостью, и **зафиксировать результат** для регрессии и отчёта.

**Компоненты (репозиторий или CI — по согласованию):**

| Компонент | Назначение |
|-----------|------------|
| Вход теста | Набор фикстур: `news_id` или текст новости + `preset_id` (домены RAG); опционально «золотой» эталон не хранится, проверяются структура, длина, ключевые поля ответа. |
| Шаги | Четыре агента LLM + детерминированный выбор якорного `source` в Chroma, как в `app/agents_pipeline.py` (или «чистые» функции для ноутбука и тестов). |
| Утверждения | HTTP/вызов функции: успешный ответ, непустые `tale`, `news_brief`, `chosen_tale_source`, `qa`; мягкая семантика: пересечение тем или эвристика по ключевым словам новости в сказке; при доступном API — расширенный прогон без мока LLM. |
| Артефакт прогона | JSON или NDJSON со **статистикой по агентам** для раздела 5 и **веб-отчёта** (раздел 6). |

## 5. Логирование и статистика по четырём агентам

**Цель:** объяснимость цепочки: что сработало при сопоставлении **новости** с **опорным произведением/чанками** из RAG и как каждый агент это отразил.

**Записи лога / статистики:**

| Агент / шаг | Что фиксировать |
|-------------|-----------------|
| **Новости (LLM 1)** | Входной сырой текст (обрезанный), выходной JSON: `summary`, `themes`, `retrieval_keywords`; хеш входа для дедупликации прогонов. |
| **RAG / выбор якоря** | Текст запроса к эмбеддеру; **top‑k** пар `(source, work_note, distance, chunk_id/индекс)`; **выбранный** `chosen_tale_source`; **вклад** каждого источника в голосование (веса `1/(1+distance)`); число чанков в промпт. |
| **Генерация (LLM 2)** | Модель, температура, оценка длины промпта/ответа (токены или символы); полный промпт в проде можно не хранить — отдельный режим «verbose» для отладки. |
| **Аудит (LLM 3)** | `approved`, `notes`, факт замены текста (да/нет), длина до/после. |
| **Вопрос–ответ (LLM 4)** | Текст вопроса, длина эталонного ответа. |

**Метрики «сходства новость ↔ сказка»:**

- Жёсткая: совпадение или близость **тем** из агента новостей с **метаданными** выбранного `source` / `heroes` (если есть в индексе).
- Мягкая: доля ключевых слов из `summary` в итоговой сказке; косинус эмбеддингов между сводкой новости и началом сказки (при отдельной компактной модели).

**Хранение:** файл сессии `stats_run_<timestamp>.json` или строка в SQLite/CSV для агрегации; персональных данных нет.

## 6. Веб-отчёт по статистике

**Цель:** одна страница (или несколько вкладок), обновляемая после прогонов e2e и ручных запусков, с **таблицами и диаграммами** по данным раздела 5.

**Содержание интерфейса:**

- Таблица прогонов: дата, новость (заголовок/ид), `chosen_tale_source`, число чанков, флаг аудита, длина сказки.

- Экспорт: CSV/JSON рядом с логами.

**Технологии:** статический HTML + Chart.js из сгенерированного JSON; либо расширение FastAPI (`GET /api/reports/...`) с шаблоном Jinja2; логин на первом этапе не обязателен.

## 8.1. Актуальные system / user промпты (`app/agents_pipeline.py`)

Источник: функция `run_four_agent_pipeline`. Ниже — **текущие** строки из кода репозитория; в user-частях указано, какие значения подставляются в рантайме.

### RAG до агентов LLM (не промпт, контекст для читателя)

- Запрос к ретриверу собирается как  
  `retrieval_hint` (из пресета) + `retrieval_keywords` + темы + сводка (`rag_query`).
- По умолчанию (`rag/rag_enhanced.py`, без `FAIRYNEWS_RAG_LEGACY=1`):
  - **гибридный поиск:** порядок чанков смешивается из dense‑ранга (L2 по
    эмбеддингам) и **BM25** по токенам запроса через **RRF** (Reciprocal Rank
    Fusion); параметр `k` в формуле `1/(k+rank)` задаётся
    **`FAIRYNEWS_RAG_RRF_K`** (по умолчанию 60).
  - **итеративное извлечение:** второй запрос = исходный текст + короткие
    снимки лучших чанков первого прохода; итоговый ранг = RRF по двум спискам
    индексов (snapshot); для **Chroma** два прохода дополнительно сливаются по
    **RRF позиций** в итоговом списке (`iterative_merge: rrf_ranks` в метаданных).
    Отключается `FAIRYNEWS_RAG_ITERATIVE=0`.
  - **сжатие контекста:** перед попаданием в `rag_block` текст чанка укладывается
    в `FAIRYNEWS_RAG_CHUNK_MAX_CHARS` (по умолчанию 900): сначала предложения
    с пересечением по темам / ключам / префиксу сводки и тексту
    `retrieval_hint` пресета (`focus_phrases`) **в порядке следования в тексте**,
    затем остальные
    предложения по порядку — без перемешивания абзаца только по «весу».
- Откат на старый путь только dense: **`FAIRYNEWS_RAG_LEGACY=1`** (как
  `retrieve_plot_records` / `retrieve_plot_records_from_snapshot`).
- В промпт агента «сказка» попадает блок `rag_block`: чанки с заголовками  
  `[domain | source | content_lang | work_note]` и разделителем `\n\n---\n\n`.

### 8.2. Переменные окружения расширенного RAG

| Переменная | По умолчанию | Смысл |
|------------|--------------|--------|
| `FAIRYNEWS_RAG_LEGACY` | выкл. | `1` — только векторный поиск, один проход, без сжатия |
| `FAIRYNEWS_RAG_ITERATIVE` | вкл. | `0` — один гибридный проход |
| `FAIRYNEWS_RAG_CHUNK_MAX_CHARS` | `900` | Бюджет символов на чанк после выжимки |
| `FAIRYNEWS_RAG_CHROMA_POOL` | `120` | Верхняя граница пула чанков из Chroma для BM25/RRF |

Поле `report.rag` в JSON прогона дополняется метаданными (`mode`,  
`hybrid_bm25_rrf`, `iterative_rounds`, `chunk_compression`, `rrf_k`,
`iterative_merge` при двух проходах, …).

### Стало ли «лучше»

Автотесты не измеряют качество сказки: они проверяют отсутствие регрессий.
Ожидаемый эффект — выше полнота по редким терминам (BM25), стабильнее якорь
при втором проходе и слиянии рангов (Chroma), читаемее выжимка чанков
(релевантные предложения в порядке текста, затем остальное). Сравнение по
смыслу — вручную на одной новости с `FAIRYNEWS_RAG_LEGACY=1` vs без неё, как
в [experiments/ETAP4_RUN_RESULTS.md](experiments/ETAP4_RUN_RESULTS.md).

### Локально: два варианта на Ollama

Проверялись **две** локальные модели в одинаковых условиях (одна и та же новость

| Вариант | Пример тега Ollama | Сильные стороны | Ограничения для пайплайна |
|---------|-------------------|-----------------|----------------------------|
| **A** | компактная 3B или близкий класс | быстрый отклик на CPU/GPU, малый объём VRAM | слабее удержание **строгого JSON** на агентах news/audit; русская сказка местами однотипнее |
| **B** | 7–8B с хорошим русским или инструкт | заметно стабильнее связный текст и темы | дольше wall-clock; на слабой машине неприемлемая задержка |

**Отбор среди локальных:** по **времени** (`report.timing` или секундомер), доле
успешных прогонов без ручной правки JSON и **субъективной** оценке сказки
(соответствие новости, народный стиль). Победитель локального тура использовался
как нижняя планка «бесплатного» контура; для сдачи диплома облачный вариант
всё равно сравнивался отдельно.

### Облако: DeepSeek, ChatGPT (OpenAI), Anthropic, Groq

Сравнение не «рейтинг моделей в вакууме», а **пригодность к четырём вызовам**
(chat + `json_object` на трёх этапах) и доступности из вашей среды.

| Кандидат | Что учитывали | Параметры / переменные в проекте |
|----------|---------------|----------------------------------|
| **DeepSeek** | цена, латентность, перенос промптов | `DEEPSEEK_API_KEY`, backend `deepseek`, модель из кабинета (например `deepseek-chat`) |
| **ChatGPT (OpenAI API)** | устойчивый JSON, качество сказки, единый стиль в uniform | `OPENAI_API_KEY`, опционально `OPENAI_API_BASE` для прокси; `OPENAI_MODEL` / uniform-переменные |
| **Anthropic** | альтернатива по политике данных и качеству | при подключении — отдельный ключ и согласование формата ответов с JSON-этапами |
| **Groq** | скорость и экономия на массовых прогонах | `GROQ_API_KEY` / `GROQ_KEY`, backend `groq`, модель из списка Groq |

Идентификаторы вида **`openai/gpt-4o-nano`** в каталогах OpenRouter-типа **не
использовались**: для «nano»-класса при необходимости берут **подтверждённый** id
(например `gpt-4.1-nano` / строка из кабинета прокси), иначе — `gpt-4o-mini` или
другой **явно перечисленный** в биллинге модельный ярус **OpenAI**.

### Шаги выбора (зафиксировать в `experiments/`)

1. **Вход:** текст новости, `preset_id`, `FAIRYNEWS_RAG_BACKEND=snapshot` (или
   согласованный режим).
2. **Режим пайплайна:** uniform (`FAIRYNEWS_LLM_UNIFORM_STAGES=1`) для сравнения
   «одна модель на все этапы»; при необходимости отдельный прогон mixed (§10.1).
3. **Смоук-тест канала** тем же клиентом, что пайплайн: `python -m app.llm_connect_try`
   (ключ и base из `.env`).
4. **Зафиксировать** для каждого кандидата: `pipeline_wall_sec`, `llm_total_sec`,
   длина сказки, заполненность `news_brief`, `qa`, флаг аудита.
5. **Субъективно:** стиль, связность с новостью, отсутствие «склеек» между
   агентами (важнее при uniform).

### Вывод по архитектуре проекта

Пайплайн из коробки опирается на **OpenAI-совместимый** вызов; для задач
**JSON + длинная генерация** наиболее предсказуемо в этом контуре семейство
**ChatGPT / OpenAI**. Остальные провайдеры остаются полезными для цены и скорости,
но **рекомендуемая база** для отчёта и демонстрации — **OpenAI-модель**
(прямой API или прокси с **проверенным** id из каталога).

### Итоговое решение (за человеком)

Финальный выбор конкретной строки модели **не автоматизирован**: его фиксирует
автор после шагов выше. **В данной работе принято решение использовать модель
линейки ChatGPT (OpenAI)** — как компромисс качества JSON, текста сказки и
воспроизводимости сдачи; конкретный id (например уровень `mini` или строка из
прокси) указывается в `.env` и в markdown-файле экспериментов без раскрытия
ключей.

Подробные таблицы прогонов — [experiments/ETAP4_RUN_RESULTS.md](experiments/ETAP4_RUN_RESULTS.md);
команды — [docs/WORKFLOW.md](docs/WORKFLOW.md).

## 10. Пять сценариев прогона с реальным LLM (стоимость ↔ качество)

Идея: **одна и та же новость и пресет RAG** (например текст из `docs/pipeline_walkthrough_news.txt` и `russian_folk`), меняется **модель/провайдер** или **параметр контура** — так проще сравнивать в файле экспериментов.

| № | Сценарий | Провайдер / модель (ориентир) | Роль в портфеле |
|---|----------|-------------------------------|-----------------|
| **1** | Максимум экономии на итерациях | **Groq** `llama-3.1-8b-instant` (или `llama-3.3-70b-versatile`, если доступна) | Очень дешёво и быстро; пригодно для массовой обкатки промптов и RAG; стиль сказки слабее. |
| **2** | Баланс цена / качество для продакшн-прототипа | **OpenAI** `gpt-4o-mini` | Устойчивый JSON от агентов 1/3/4 и связный текст; хороший дефолт для отчёта. |
| **3** | Акцент на русском и «гладкой» сказке | **GigaChat** `GigaChat` или линейка **Pro/Max** (по лицензии кабинета) | Оплата в ₽, сильный русский; удобно для дипломной демонстрации в РФ. |
| **4** | Верхняя планка качества (точечно) | **OpenAI** `gpt-4o` *или* топовая модель GigaChat | Дороже; 1–2 прогона для «лучшего варианта» в итогах экспериментов. |
| **5** | Изоляция RAG (модель не меняется) | Та же модель, что в №2 или №3 | Например: `FAIRYNEWS_RAG_BACKEND=snapshot` vs Chroma с полным индексом; или `k=8` vs `k=15`; или пресет `russian_folk` vs `default`. Показывает вклад ретрива без шума от смены LLM. |

**Замечание:** если какой-то API недоступен (регион, баланс), вариант помечается в эксперименте как «не выполнен» с причиной — это нормальная часть итогового вывода.

## 11. Дополнительные улучшения (RAG, workflow, интерфейс)

**RAG**

- Лог **полного** `rag_query` и **среза** текстов top‑k в отчёте (уже частично есть) + опция «склеить чанки одного `source`» для читабельности.
- Лёгкий **MMR / разнообразие источников** в top‑k, чтобы не залипать на одном произведении.
- Явное **ограничение длины** RAG-блока в промпте генерации (обрезка по символам) при скачках объёма корпуса.

**Workflow**

- Маленький CLI или Makefile-цели: `run-e2e`, `capture-screens`, `regen-notebook-deps` — меньше ручных шагов перед сдачей.
- Версионирование пресетов и промптов (один YAML или модуль констант) — проще воспроизводить «Вариант №2» через год.

**Интерфейс**

- **Индикатор загрузки** на шаге 3 и блокировка повторного нажатия «Сгенерировать» до ответа.
- Копирование **run_id** / ссылки на отчёт в буфер; предпросмотр **первых N символов** сказки в сводке списка отчётов.

### 12.1. Ключи `.env` и пять прогонов (§10)

В текущем `.env` заданы **три** провайдера: `GIGACHAT_API_KEY`, `GROQ_KEY`, `DEEPSEEK_API_KEY` (**без** строки `OPENAI_API_KEY`).

| Сценарий §10 | Как закрыть сейчас | Рекомендация |
|--------------|-------------------|--------------|
| 1 Groq | `GROQ_KEY` | Ок; при 403 проверьте ключ и список моделей на кабинете Groq. |
| 2 **4o-mini** | Нет ключа | Добавьте локально **`OPENAI_API_KEY`** (+ при необходимости `OPENAI_MODEL=gpt-4o-mini`). |
| 3 GigaChat | `GIGACHAT_API_KEY` | Ок; scope при необходимости `GIGACHAT_SCOPE` (см. `app/gigachat_requests.py`). |
| 4 **4o** | Нет ключа | Локально: **`OPENAI_API_KEY`** и **`OPENAI_MODEL=gpt-4o`** для 1–2 дорогих прогонов *или* топ-модель GigaChat, если доступна. |
| 5 только RAG | Любой работающий LLM из таблицы | Меняйте только `FAIRYNEWS_RAG_BACKEND` / `k` / пресет. |

Итого: для полного покрытия плана из пяти сценариев **желательно добавить OpenAI-ключ** локально (не коммитить). До этого сценарии 2 и 4 в файле экспериментов помечайте как «не выполнено (нет ключа)» или замените на дополнительные сравнения **DeepSeek vs Groq vs GigaChat** при одинаковом RAG.

**Per-stage и замеры:** переменные `FAIRYNEWS_STAGE_*`, uniform `FAIRYNEWS_LLM_UNIFORM_STAGES`, метрики времени в `report.timing` — см. [docs/WORKFLOW.md](docs/WORKFLOW.md) и [experiments/ETAP4_RUN_RESULTS.md](experiments/ETAP4_RUN_RESULTS.md).

## 4. Технологии: как набрать достаточно информации для RAG

**Сказки (относительно статичный корпус)**

- **Загрузка:** прямая выгрузка текстовых форматов (txt, html, fb2→текст) там, где это разрешено; для **Project Gutenberg** — зеркала и каталог по ID книги. В коде проекта загрузка Gutenberg — **httpx** (см. `rag/gutenberg.py`); конфиг семян — **YAML**; индекс — **ChromaDB**; эмбеддинги — **sentence-transformers**. Для **парсинга сайтов** при расширении корпуса: **httpx** + при необходимости **BeautifulSoup** / **lxml**, с соблюдением **robots.txt** и лицензий.
- **Сегментация:** разбиение на **чанки** (например 400–900 токенов или по абзацам/главам) с перекрытием; сохранение **метаданных**: язык, регион/тип («русская», «европейская», «восточная»), источник, название сказки.
- **Эмбеддинги:** одна модель эмбеддингов для русского и английского (мультиязычная, напр. семейство **sentence-transformers** / API провайдера) либо **разные** энкодеры только если эксперимент покажет выигрыш — чаще хватает одной мультиязычной.
- **Индекс:** векторное хранилище (**Qdrant**, **Chroma**, **LanceDB**, **pgvector** в PostgreSQL) + при необходимости **полнотекстовый** индекс (**BM25** в Elasticsearch/OpenSearch или **FTS5** в SQLite) для гибридного поиска.
- **Гибридный поиск:** dense (семантика) + sparse (ключевые слова, имена героев) + **reranker** (кросс-энкодер) на top-k — повышает полноту для «редких» мотивов сказки.

**Новости (динамический поток)**

- **Официальные каналы:** **RSS/Atom** где доступны; **API** агрегаторов (если есть ключ и условия).
- **Парсинг HTML:** только с соблюдением правил сайта; лучше **планировщик** (cron/Celery/APScheduler), **дедупликация** по URL и по тексту (SimHash), **нормализация** даты и языка.
- **Сжатие под RAG:** не хранить полный сайт — **заголовок + лид + 1–3 абзаца** или автоматическая **экстрактивная суммаризация** малой моделью; для политики важны **дата** и **источник** в метаданных чанка.
- **Актуальность:** отдельный **индекс «новости»** с TTL или версионированием по дням, чтобы не смешивать устаревшее с «хвостом» старых новостей без явного запроса.

**Общее**

- **Оценка качества:** периодический **recall@k** на ручном наборе запросов; логирование запросов и retrieved chunks.
- **Объём «достаточно»:** ориентир — **тысячи** чанков по сказкам (после дедупа) и **регулярное пополнение** новостей; точный порог — по метрикам качества генерации, а не по числу сайтов.

## 5. Один или несколько RAG? Один или несколько агентов?

**RAG**

- **Рекомендация для проекта:** **несколько логических RAG-контуров** (отдельные коллекции/индексы или одна коллекция с жёсткими **фильтрами по метаданным**): `russian_tales`, `european_american_tales`, `eastern_tales`, `news_current`.
- **Почему не один «общий котёл»:** разные типы текстов и разная **актуальность** (новости vs вековые сказки); проще настраивать chunking, веса гибридного поиска и **политику отбора** (например, для сюжета брать 2 чанка из новостей и 5 из русских сказок).
- **Одна модель эмбеддингов** на все коллекции обычно проще; **несколько индексов** не равны «нескольким нейросетям генерации».

**Агенты (согласовано с этапом № 1)**

- **Базовый вариант для проекта «Нейро-сказочник»:** **мультиагентный пайплайн** под оркестрацией FastAPI: роли **новости → генерация сказки (с RAG по похожим сюжетам) → аудит → вопрос и эталонный ответ**; **не менее двух** вызовов GPT, фактически **отдельный промпт и инструкция** на каждый шаг.
- **Модели:** разные агенты могут вызывать **разные** endpoints/модели — выбор уточняется **анализом стоимости и качества**; это не то же самое, что «отдельная LLM на каждый **веб-сайт** источника».
- **Расширение:** глубокий **агентный режим** с циклами tool-use («подумал — запросил ещё документы») — по необходимости поверх линейной цепочки.

**Краткий итог:** **несколько RAG-контуров** + **оркестратор** + **несколько вызовов GPT с роль-специфичными промптами** (мультиагентная схема); упрощённая схема «один запрос LLM на всё» для этого проекта **не целевая**. См. `etap1_utverzhdenie_temy.md`.

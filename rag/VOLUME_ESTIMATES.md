# Оценка объёма корпуса и чанков (сказки, RAG без новостей)

Политические **новости в этот индекс не входят** — они обслуживаются **агентом
новостей** и конвейером данных вне RAG.

## Что уже заложено в коде (стартовые семена)

Файл `rag/sources/fairy_tale_seeds.yaml`: **12** полнотекстов Project Gutenberg
(европейский/ориентальный блок + русский фольклор в англ. PD + Толстой
**38025** + локальные маски `soviet/`, `russian/`, общий `local_tales/`).
Число чанков после `python -m rag --reset` зависит от сети и локальных файлов;
ориентир для прежнего набора из 7 книг был **3980** чанков — после добавления
русских ebooks счётчик вырастет (пересоберите индекс для актуальной цифры).

## Плановый корпус «классика + современность»

| Слой | Ориентир числа произведений / сборников | Объём текста | Оценка чанков |
|------|----------------------------------------|--------------|---------------|
| Русские тексты (ФЭБ, lib.ru, ручная выгрузка, локальные `.txt`) | 300–900 | ~15–60 Mb | **8 000–35 000** |
| Европейские PD (расширение Gutenberg + архивы) | 200–600 | ~10–40 Mb | **6 000–28 000** |
| Американский фольклор в PD | 50–200 | ~3–15 Mb | **1 500–8 000** |
| Восточные циклы («1001 ночь» и др. в PD) | 80–250 | ~8–35 Mb | **4 000–22 000** |
| Современные лицензированные / ручные вставки | по проекту | переменно | +5–25% к слою |

**Итого после наполнения (порядок величины):** **20 000–90 000** чанков при
текущих `CHUNK_TARGET_CHARS=900` / `OVERLAP=120`, либо **15 000–65 000**, если
агрессивнее дедуплицировать повторяющиеся издания.

## Связь с агентами

- **Агент генерации сказки** использует этот индекс + промпт + контекст от
  **агента новостей** (новости **не** из RAG).
- Далее по цепочке: **агент аудита**, **агент «вопрос–ответ»** (отдельные вызовы
  GPT с отдельными инструкциями).
- **Мультиагентная оркестрация** (FastAPI) объединяет роли; **несколько
  RAG-контуров** в перспективе можно разнести по коллекциям или фильтрам
  `domain`; сейчас одна коллекция `fairy_tales_plots` с метаданными `domain`.

## Отчёт по индексу

```bash
python -m rag.report
python -m rag.report --lang ru
python -m rag.report --lang en
python -m rag.report --lang bi
python -m rag.report --show-chunks-per-work
python -m rag.report --json
```

Раздел «чанков на произведение» в текстовом отчёте по умолчанию скрыт; в JSON
поле `chunks_per_work` сохраняется.

По умолчанию **русский текст с английским в скобках** (`--lang ru`). В JSON
есть `notes.ru`, `notes.en`, **`by_content_lang`**, **`summary`**
(кратко: языки, топ стран, авторов, образов героев).

В отчёте: число **чанков**, число **работ** (уникальный `source`: Gutenberg id
или путь к `.txt`), распределение по **domain**, **country**, **author**,
**content_lang** (язык текста чанка), подсказки по **heroes** (строка из YAML,
грубая токенизация по `;` / `,`).

Коды **RU** (русский слой: Gutenberg Afanasyev и др., папка `russian/*.txt`) и
**USSR** (папка `soviet/*.txt`) появляются **только после** того, как эти
источники попали в индекс. Если в отчёте их нет — выполните полную пересборку:
`python -m rag --reset` (и при необходимости положите `.txt` в подпапки).

## Периодическое обновление (пайплайн)

Инкремент: для каждого источника считается **SHA-256** полного текста; если
хеш совпал с сохранённым в `data/rag_pipeline_state.json` — источник
пропускается; иначе старые чанки удаляются (`delete` по `source`) и
добавляются новые.

```bash
python -m rag.pipeline update
```

Фоновый цикл (простой `sleep`, без внешнего cron):

```bash
python -m rag.pipeline daemon --interval-hours 24
```

Для **первого** запуска после изменения схемы метаданных (author, country,
heroes) надёжнее выполнить полную пересборку: `python -m rag --reset`, затем
удалить `data/rag_pipeline_state.json`, чтобы пайплайн пересчитал хеши.

Планировщик ОС (пример Windows **Task Scheduler**): периодический запуск
`python -m rag.pipeline update` из каталога проекта с активированным venv.
